# What we do next re OpenClaw Autopilot?

This plan is generated from the SCQH here: https://github.com/datopian/openclaw-aaas/blob/main/docs/scqh/2026-03-19.md

> **Autopilot as OSS-backed prototype + bounded learning experiment, not product launch**

Assuming the goals for these 2+ weeks are:

1. lock the positioning
2. set up the bounded experiment
3. publish enough OSS/content to support credibility
4. create a lightweight funnel for real demand
5. avoid drifting into bespoke chaos

---

# How to read this plan

This document covers two layers:

1. the **first 2 weeks**, where the goal is to get the experiment set up properly
2. the **rest of Q2**, where the goal is to test real demand without drifting into bespoke work

The first 2 weeks are not the quarter's outcome.
They are the setup phase for the quarter's learning experiment.

---

# 2-week objective

By **April 6**, Datopian should have:

* a clear public positioning statement
* a tightly scoped service offer
* one cleaned-up OSS asset published
* one piece of content published
* one simple landing / inquiry path
* one internal learning loop in place
* clear operating constraints for the experiment

Success is **not** measured by revenue in these 2 weeks.
Success is measured by **readiness to run a disciplined experiment**.

---

# Q2 north star

Use a demand-validation metric, but do **not** use raw bid volume as the north star.
Raw bid count is too easy to game and creates pressure to chase bad-fit work.

The recommended north star for Q2 is:

* **number of qualified, in-scope bids / proposals submitted to real demand opportunities**

This keeps the metric concrete and controllable, while still reinforcing the strategic goal from the SCQH: learn from real demand without becoming a generic AI agency.

## What counts as a qualified bid

A bid only counts if **all** of the following are true:

* the work is clearly about setup, deployment, debugging, stabilization, or light customization
* the project can plausibly be delivered as a fixed-scope engagement
* the client brief is concrete enough to judge fit and learn from
* the work fits inside the **0.25-0.5 FTE** experiment cap
* the work does **not** require a bespoke product build, broad strategy engagement, or open-ended implementation

## Thresholds for Q2

* **Below 10** qualified bids: insufficient signal; likely too little demand or too weak a funnel
* **10 qualified bids**: minimum viable experiment threshold
* **15 qualified bids**: success
* **20 qualified bids**: stretch / exceptional result, but only if quality and scope discipline hold

## Supporting outcome checks

The north star should be paired with a small set of guardrail checks so the team does not optimize for volume alone:

* at least **3-5 meaningful responses or conversations**
* at least **1-3 paid engagements or strongly qualified near-misses**
* at least **5 reusable patterns / playbooks / checklists** captured from market feedback or delivery work
* **0 accepted out-of-scope bespoke projects**

The important critique is this: **"20 bids" is not a win if they are low-fit bids that teach you nothing.**
Quality-adjusted demand is the right measure here, not raw activity.

---

# Guardrails for the whole 2 weeks

These need to be explicit from day 1.

## Non-goals

* no full product launch
* no broad “AI agency” positioning
* no custom work outside the narrow experiment scope
* no major platform engineering effort
* no polishing for polish’s sake

## Hard limits

* max **0.25–0.5 FTE**
* only accept work in:

  * setup
  * debugging
  * light customization
* fixed-scope engagements only
* every activity must contribute to:

  * learning
  * authority
  * demand validation
  * reusable assets

---

# What makes this handoff-ready

The strategy is sound, but the document should be treated as an execution brief only if the following are made explicit before handoff:

* one **single accountable owner** for the experiment
* one named owner for each major deliverable: strategy note, offer, OSS repo, content, landing page, proposal templates, learning system
* the exact location where each asset lives: repo path, docs folder, landing page destination, tracker, pattern library
* a weekly review slot with one decision-maker who can enforce the guardrails
* a definition of done for each major deliverable so the assignee does not over-polish

If those points stay implicit, a teammate can execute the tasks but will still have to guess at priorities and tradeoffs.

## Minimum implementation cadence for the rest of Q2

Once the setup phase is done, run the experiment on a simple weekly rhythm:

1. **Monday:** identify good-fit opportunities, send qualified bids, update the tracker
2. **Midweek:** review replies, refine messaging, reject bad-fit work quickly
3. **Friday:** capture learning, update the pattern library, and check for scope drift

This cadence matters because the real output is not only proposals sent.
It is also the learning loop that turns demand into reusable knowledge.

# Week 1: Foundation and packaging

## March 24–30

## Day 1 — Tuesday, March 24

### Goal

Align internally on positioning and experiment rules.

### Tasks

* Finalize the core positioning:

  * **Autopilot is an open reference implementation and learning vehicle**
  * Datopian is a **specialized setup / AgentOps partner**
* Decide the exact experiment boundaries:

  * channel(s): Upwork only, or Upwork + direct inbound
  * offer scope
  * team allocation
  * review date after 6–8 weeks

  * what types of work are explicitly out of scope

### Deliverables

* 1-page internal strategy note
* experiment charter with guardrails
* list of in-scope / out-of-scope work

### Decision to lock

Pick **one label** for the category you’ll use consistently.
My recommendation: **AgentOps** externally, with “AI SRE / agent operations” as supporting language.

---

## Day 2 — Wednesday, March 25

### Goal

Define the actual offer.

### Tasks

Package the service into **1–2 very simple offers**, not a menu of ten things.

### Recommended offers

#### Offer 1: Agent Setup Sprint

* deploy on client infra
* basic config
* basic tool wiring
* opinionated defaults
* limited debugging
* handoff notes

#### Offer 2: Agent Debug / Stabilization

* diagnose broken or flaky setup
* fix key issues
* recommend configuration and operational changes

### Also define

* target customer profile
* ideal project size
* pricing band
* exclusions

### Suggested pricing

Keep it simple:

* **Setup Sprint:** $500–$1,000 fixed
* **Debug/Stabilization:** $300–$800 fixed
* optional maintenance only if extremely lightweight

### Deliverables

* service description
* pricing guidance
* short qualification checklist

Start outbound experimentation carefully.

### Tasks

* apply only to tightly matching Upwork jobs
* max 3–5 proposals
* reject vague “build me AI startup” work

### Screening criteria

Accept only jobs that fit:

* setup
* deployment
* debugging
* light customization

Reject:

* full product builds
* undefined strategy work
* large integrations

* first batch of targeted proposals sent

## Day 14 — Monday, April 6

### Goal

Run the two-week review and decide next step.

### Review questions

1. Did we establish a clear non-product position?
2. Did we create a credible OSS + content footprint?
3. Did we see real qualified demand?
4. Did we stay within scope?
5. Did we generate reusable knowledge?
6. Is this worth continuing for the next 4–6 weeks?

### Decision options

#### Option A: Continue

Criteria:

* at least some qualified demand
* clear learning value
* manageable opportunity cost
* no major drift into bespoke work

#### Option B: Continue, but narrower

Criteria:

* demand exists but is messy
* messaging too broad
* need tighter scope

#### Option C: Stop

Criteria:

* low-quality demand only
* no reusable learning
* too distracting relative to core work

### Deliverables

* 2-week review memo
* go / narrow / stop decision
* next-phase hypothesis

# What to prioritize if time gets tight

If you cannot do everything, do these five first:

1. **Finalize positioning and guardrails**
2. **Package the service offer**
3. **Publish one OSS asset**
4. **Publish one strong article**
5. **Start a small number of tightly scoped proposals**

Everything else is secondary.

* 1 landing page

* 1 article
* 1 video

* 3–4 internal templates
